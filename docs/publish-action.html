NPM Publish Action

Detailed analysis and configuration code paths of the package publishing pipeline.

1. Trigger Conditions

The workflow is configured in publish-and-notify.yml to run under two conditions:

  • Release Trigger: Automatically runs when a release is successfully published (release: types [published]).
  • Manual dispatch: Can be manually triggered by a repository maintainer from the Actions interface (workflow_dispatch).

2. Execution Jobs Breakdown

The workflow contains two primary sequential jobs: publish and notify-dependents.

Job 1: Publish (runs on ubuntu-latest)

Establishes the runtime environment, runs tests, and deploys the build artifact to npmjs:


  • Checkout repository: Fetches the source code using actions/checkout@v4.
  • Setup Node.js: Configures Node version 20 and sets the default registry url to https://registry.npmjs.org using actions/setup-node@v4. This automatically configures authentication maps for NPM publish.
  • Install dependencies: Executes npm ci for clean, repeatable dependency builds.

3. Version Safety Publish Script

To avoid build errors if the current local version is already published on the NPM registry (such as running the action on an old branch), the publish step evaluates the NPM registry beforehand:

bash
# Step: Publish to npm PACKAGE_NAME=$(node -p "require('./package.json').name") PACKAGE_VERSION=$(node -p "require('./package.json').version") # Run query against the registry to see if version exists if npm view "${PACKAGE_NAME}@${PACKAGE_VERSION}" version >/dev/null 2>&1; then echo "Version ${PACKAGE_VERSION} of ${PACKAGE_NAME} is already published on npm. Skipping publish." else # Only publish if registry query exits with non-zero (404 Not Found) npm publish fi

How it works:

1. `node -p` reads the `name` and `version` fields directly from the local `package.json`.
2. `npm view @ version` checks if the specified version is indexed in the NPM registry. If the version exists, it returns exit code `0` (success). If the version does not exist, it returns exit code `1` (fail/not found).
3. The conditional logic skips publishing on exit code `0`, avoiding redundant publish attempts and preventing workflow failure.

4. Calling Notify Dependents

Once the publication job finishes successfully (or skips safely), the second job notify-dependents executes. It uses GitHub Actions reusable caller syntax to forward permissions:

yaml
notify-dependents: needs: publish uses: ./.github/workflows/notify-dependents.yml secrets: REPO_DISPATCH_TOKEN: ${{ secrets.REPO_DISPATCH_TOKEN }}
