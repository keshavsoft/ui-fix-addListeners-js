Notify Dependents Action

Detailed analysis of the post-publish repository dispatch notification mechanism.

1. Reusable Workflow Architecture

The notify-dependents.yml action is built as a reusable callee workflow. It is designed to receive a Personal Access Token (`REPO_DISPATCH_TOKEN`) from the caller publish workflow to authorize API requests against external repositories.

yaml
on: workflow_call: secrets: REPO_DISPATCH_TOKEN: required: true

2. Registry Replication Delay

When a package is pushed to npmjs, replication latency across regional registries and CDNs can range from 5 to 15 seconds. If a downstream repository immediately runs an update trigger, it may fetch the old version.

To avoid replication race conditions, we pause the step for 15 seconds:

yaml
- name: Wait for NPM registry replication run: sleep 15

3. API Dispatch Payload & Headers

Once the replication delay concludes, the action fires a repository dispatch webhook to the downstream dependent project kschema-fs-api-gen-get-actions:

bash
curl -X POST \ -H "Accept: application/vnd.github+json" \ -H "Authorization: Bearer ${{ secrets.REPO_DISPATCH_TOKEN }}" \ https://api.github.com/repos/keshavsoft/kschema-fs-api-gen-get-actions/dispatches \ -d '{"event_type":"dependency-updated"}'

Breakdown of the Webhook Parameters:

  • -H "Accept: application/vnd.github+json": Instructs the GitHub API to return answers in the standard JSON format.
  • -H "Authorization: Bearer ...": Attaches the Personal Access Token (PAT) required to authorize dispatch actions.
  • -d '{"event_type":"dependency-updated"}': Sends a JSON payload containing the event identifier. Downstream workflows must register this exact key (dependency-updated) under their dispatch conditions to trigger updates.
